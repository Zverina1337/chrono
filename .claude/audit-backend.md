# Аудит Backend (Rust/Tauri) — Chrono

## CRITICAL — ломает функциональность или опасно

### 1. `project_repo.rs` — ручной расчёт даты в now()
- **Строки:** функция `now()`, ~40 строк
- **Проблема:** Ручная арифметика для вычисления UTC-даты из `SystemTime` вместо крейта `chrono`
- **Последствие:** Хрупкий код, сложно поддерживать, потенциальные баги с високосными годами и edge cases
- **Направление:** Подключить крейт `chrono` и заменить на `Utc::now().format(...).to_string()`
- [ ] Исправлено

### 2. Mutex блокирует все запросы
- **Файлы:** все `*_cmd.rs`
- **Проблема:** Один `Mutex<Connection>` на всё приложение, каждая команда делает `db.lock().unwrap()`
- **Последствие:** Все запросы выполняются последовательно, UI может фризить при тяжёлых операциях
- **Направление:** Рассмотреть connection pool (r2d2) или async Mutex (tokio), или хотя бы минимизировать время удержания lock
- [ ] Исправлено

## HIGH — нужно исправить

### 3. Нет валидации входных данных в commands
- **Файлы:** все `*_cmd.rs`
- **Проблема:** Команды передают данные напрямую в repository без проверки
- **Последствие:** Пустые строки, слишком длинные названия, невалидные UUID проходят в БД
- **Направление:** Добавить слой валидации в commands или создать отдельный validation модуль
- [ ] Исправлено

### 4. `create_project()` нарушает SRP
- **Файл:** `project_repo.rs`
- **Проблема:** Функция создаёт проект И дефолтные статусы ("Бэклог", "В работе", "Готово") И дефолтные приоритеты ("Низкий", "Средний", "Высокий")
- **Последствие:** Невозможно создать проект без дефолтов, сложно тестировать изолированно
- **Направление:** Вынести создание дефолтов в отдельную функцию или сервис
- [ ] Исправлено

### 5. AppError слишком простой
- **Файл:** `error.rs`
- **Проблема:** Только два варианта: `Database` и `NotFound`. Serialize превращает всё в строку
- **Последствие:** Фронтенд не может различить типы ошибок (validation vs not found vs db error)
- **Направление:** Добавить варианты `Validation`, `Conflict`, `Internal`. Сериализовать как `{ code, message }`
- [ ] Исправлено

### 6. Дублирование паттерна позиции
- **Файлы:** `task_repo.rs`, `status_repo.rs`, `priority_repo.rs`, `subtask_repo.rs`
- **Проблема:** `COALESCE(MAX(position), -1)` копипастится в каждом repo
- **Последствие:** Нарушение DRY, при изменении логики нужно править в 4+ местах
- **Направление:** Извлечь в утилитную функцию `next_position(conn, table, filter_col, filter_val)`
- [ ] Исправлено

## MEDIUM — улучшения качества

### 7. Format string в SQL запросах
- **Файлы:** `task_repo.rs`, `time_entry_repo.rs`
- **Проблема:** `format!("SELECT {SELECT_COLS} FROM tasks WHERE uuid = ?1")` — используется `format!` без необходимости
- **Последствие:** Визуально похоже на SQL injection pattern, хотя параметры подставляются безопасно
- **Направление:** Использовать `const` строки или `concat!` макрос
- [ ] Исправлено

### 8. N+1 в create_project()
- **Файл:** `project_repo.rs`
- **Проблема:** Дефолтные статусы и приоритеты вставляются в цикле отдельными INSERT
- **Последствие:** 7 отдельных запросов вместо 1-2 batch insert
- **Направление:** Объединить в один INSERT с множественными VALUES
- [ ] Исправлено

### 9. Нет транзакций для составных операций
- **Файлы:** `project_repo.rs`, `task_repo.rs`
- **Проблема:** Составные операции (создание проекта + статусы + приоритеты) не обёрнуты в транзакцию
- **Последствие:** При ошибке на середине данные останутся в inconsistent state
- **Направление:** Использовать `conn.execute_batch("BEGIN; ... COMMIT;")` или `Transaction`
- [ ] Исправлено

### 10. `Option<Option<String>>` в UpdateTask
- **Файл:** `models/task.rs`
- **Проблема:** Паттерн `Option<Option<String>>` для nullable полей — awkward и неочевидный
- **Последствие:** `None` = не менять, `Some(None)` = обнулить, `Some(Some(v))` = установить. Сложно читать
- **Направление:** Создать enum `UpdateField<T> { Skip, SetNull, Set(T) }` или использовать другой подход
- [ ] Исправлено

### 11. Нет индексов в миграциях
- **Файл:** `db/migrations.rs`
- **Проблема:** Таблицы создаются без индексов кроме PRIMARY KEY
- **Последствие:** Запросы по `project_uuid`, `status_uuid`, `task_uuid` будут медленными при росте данных
- **Направление:** Добавить миграцию с CREATE INDEX для FK-колонок
- [ ] Исправлено

- **Файл:** `subtask_repo.rs`
- **Проблема:** SQLite хранит boolean как i32, конвертация вручную
- **Последствие:** Неочевидный код, возможны баги при `!= 0` vs `== 1`
- **Направление:** Создать хелпер `fn bool_from_i32(v: i32) -> bool`
- [ ] Исправлено

### 15. Нет bulk операций

- [ ] Исправлено

### 16. Boilerplate в commands
- **Файлы:** все `*_cmd.rs`
- **Проблема:** 30 команд с практически одинаковым кодом: lock mutex → call repo → return
- **Направление:** Создать макрос `tauri_command!` для генерации boilerplate
- [ ] Исправлено
