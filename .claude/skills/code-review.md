# Skill: code-review

Проведи глубокий анализ кодовой базы проекта Chrono. Проверяй как frontend (Vue 3 + TypeScript), так и backend (Rust/Tauri).

## Что проверять

### 1. Баги и ошибки
- Некорректное использование реактивности Vue (ref, shallowRef, computed — доступ через `.value`)
- Отсутствие `:key` в `v-for`
- Unsafe операции (JSON.parse без типов, `.unwrap()` без обработки)
- Утечки памяти (event listeners без cleanup, незакрытые ресурсы)
- Race conditions (Mutex, async операции)
- Некорректная работа с nullable полями

### 2. SOLID принципы
- **SRP:** каждая функция/компонент/модуль отвечает за одну задачу
- **OCP:** код открыт для расширения, закрыт для модификации
- **LSP:** подтипы могут заменять базовые типы без нарушения контракта
- **ISP:** интерфейсы не должны заставлять реализовывать ненужные методы
- **DIP:** зависимости от абстракций, а не от конкретных реализаций

### 3. Чистота кода
- Дублирование кода (DRY) — одинаковые паттерны, скопированные между файлами
- Мёртвый код (неиспользуемые переменные, импорты, функции)
- Magic strings/numbers — захардкоженные значения без констант
- Именование — понятные имена переменных, функций, компонентов
- Размер функций — функции не должны быть слишком длинными
- Вложенность — избегать глубокой вложенности (> 3 уровня)

### 4. TypeScript строгость
- Использование `any` — должен быть минимизирован
- Отсутствие типов у параметров и возвращаемых значений
- Type assertions без необходимости
- Правильное использование интерфейсов и типов

### 5. Vue-специфичное
- Props validation
- Правильное использование Composition API
- Корректность `<script setup>`
- Утечки в watchers/computed
- Правильное использование Pinia stores

### 6. Rust-специфичное (backend)
- `.unwrap()` вместо proper error handling
- Отсутствие валидации входных данных в commands
- SQL injection risks
- Конкурентный доступ к данным
- Эффективность запросов (N+1, отсутствие индексов)

## Формат вывода

Группируй результаты по severity:

### CRITICAL — баги, которые ломают функциональность
### HIGH — проблемы, которые нужно исправить в ближайшее время
### MEDIUM — улучшения для качества кода
### LOW — рекомендации и стилистические замечания

Для каждой проблемы указывай:
- **Файл и строку**
- **Описание проблемы**
- **Подсказку по исправлению** (не готовый код, а направление)
